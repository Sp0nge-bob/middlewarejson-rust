# Техническое задание: Middleware-агент для модификации JSON-подписки 3x-ui

## 1. Контекст системы

Система работает как прослойка (proxy-middleware) между панелью 3x-ui и VPN-клиентами.

**Цель:** упростить и централизовать модификацию JSON-конфига подписки — любые изменения, которые 3x-ui не поддерживает из коробки. Агент получает «сырой» конфиг от панели, прогоняет через цепочку трансформаций и отдаёт клиенту готовый результат.

Балансировщик нагрузки — **один из частых сценариев**, но не единственная задача. Платформа должна позволять добавлять новые виды модификаций без переписывания ядра.

**Инфраструктура:**

- Панель 3x-ui — отдаёт сырой JSON по HTTP(S).
- Nginx — reverse proxy, внешний HTTPS.
- Агент (Python/FastAPI) — fetch, validate, transform, respond.

**Формат подписки:** Xray-core (`protocol`, `streamSettings`, `routing`). Ответ может быть одним объектом или массивом конфигов (по одному на inbound/протокол).

## 2. Архитектурная схема

```
VPN Client → Nginx (HTTPS) → Middleware Agent → 3x-ui (локально)
                                    ↓
                            Transformer pipeline
                            (passthrough → balancer → …)
```

## 3. Требования к агенту

### 3.1. Базовый endpoint

`GET {UPSTREAM_JSON_PATH}/<sub_id>` (+ `HEAD`)

1. Запрос к upstream: `{UPSTREAM_BASE_URL}{UPSTREAM_JSON_PATH}/<sub_id>`.
2. Парсинг и валидация JSON (объект или массив; наличие `outbounds`).
3. Прогон через pipeline трансформаций (конфигурируемый).
4. Возврат модифицированного JSON с пробросом заголовков 3x-ui (`Subscription-Userinfo`, `Profile-Update-Interval` и др.).

### 3.2. Слой трансформаций

Каждый трансформер — отдельный модуль с единым интерфейсом:

```python
def transform(payload: dict | list[dict]) -> dict | list[dict]: ...
```

Трансформеры включаются/выключаются через конфиг. Phase 1 — `PassthroughTransformer` (без изменений). Далее — по мере необходимости.

### 3.3. Примеры трансформаций

| Сценарий | Описание | Приоритет |
|----------|----------|-----------|
| **Балансировщик** | Merge массива конфигов → один конфиг с уникальными тегами outbound'ов + `routing.balancers` | Высокий (частый кейс) |
| **Merge конфигов** | Объединить N отдельных JSON в один без балансировки | Средний |
| **Routing rules** | Добавить/переопределить правила маршрутизации | Средний |
| **Фильтрация нод** | Убрать или оставить outbound'ы по правилам (тег, remarks, протокол) | Низкий |
| **Переименование тегов** | Нормализация `tag` для совместимости с клиентом | Низкий |

### 3.4. Пример: балансировщик (Xray)

Не sing-box — в Xray балансировка через `routing`:

```json
"routing": {
  "balancers": [{
    "tag": "balancer",
    "selector": ["vless-1", "vless-2", "vless-3"],
    "strategy": { "type": "roundRobin" }
  }],
  "rules": [{
    "type": "field",
    "network": "tcp,udp",
    "balancerTag": "balancer"
  }]
}
```

Outbound'ы с уникальными тегами (`vless-1`, …) формируются из proxy-блоков исходного массива конфигов.

## 4. Фазы разработки

**Phase 1** — прозрачный proxy + интерфейс трансформеров + тесты.

**Phase 2+** — конкретные трансформеры по приоритету (балансировщик первым среди модификаций).
